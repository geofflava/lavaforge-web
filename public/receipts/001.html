I ran my scanner on my own agent first

2026-07-07 · runworthy 0.2.0 · detectors: gitleaks 8.30.1 · osv-scanner 2.4.0 · skillspector 2.3.9

Runworthy is my agent operations scanner: it reads a repo and answers the question you actually have — is this safe to run, and what do I fix first? This week I pointed it at two real agent repos, mine included. Both came back PROVISIONAL, the grade that means unknowns, not failures (the other two are GO and NO-GO). Here's what the reports said.

My own repo, first

Research-Cascade is my open-source research engine — an MCP server with a knowledge graph and trust scoring. The first scan (report) came back PROVISIONAL with a short, specific list:

  • Update eleven packages. Eleven dependencies in the engine's npm lockfile had open security advisories — hono alone carried twenty, and the one against vitest was rated critical. An agent inherits every one. This was the one hard, confirmed gap — not a judgment call, a fact. Straight from the report:

    Fix: Run your package manager's update or audit-fix command to bring these libraries to patched versions, starting with hono since it has the most advisories. Re-scan afterward to confirm the vulnerabilities are cleared.

  • Answer ten questions. A named owner, per-agent credentials, a tested kill-switch, an incident runbook: the ten non-negotiables a scan can't read from code. Runworthy asks them in plain English after the scan and folds the answers into the grade. Honest answer at the time: most were "not yet."

So I did the work instead of writing about it. The eleven packages: updated, re-scanned, clean — the lockfile and the manifest check both quiet. The ten questions: I built what they ask for rather than just attesting — a kill-switch that refuses every tool call until a human clears it, a human approval gate on the destructive actions (with soft-delete, so a rejected finding is recoverable, not gone), a replayable action log, per-role tool limits, an incident runbook. Each one is a commit, not a checkbox.

The part I didn't plan for: before I trust my own work I run an adversarial pass that tries to disprove every "done." It caught two controls I'd written up as finished but hadn't actually wired — an allowlist that was never loaded, a breaker that was never called. Documented-but-not-real is the precise failure this whole product exists to catch, and it caught me. I fixed both, then re-checked. The dependency gap is closed and the ten now have real answers behind them; the grade followed, but the grade was never the point — the point is it found the actual work and told me what to do first.

What it didn't flag: one detector found six prompt-injection strings in the repo — all in test files, because Research-Cascade tests its own injection defenses. The pipeline treats test fixtures as non-evidence, so none of them turned into an assessment.

The false alarm it didn't raise

The second repo is a widely used open-source research agent. I'm not naming it or publishing its report — house policy until a maintainer's seen it. Same shape as mine: PROVISIONAL, four dependencies with open advisories (two rated critical), and two code paths flagged as leads worth an engineer's hour.

One detector flagged a "generic API key" in that project's .env.example, the template with empty OPENAI_API_KEY= lines you're meant to fill in. Plenty of tools would take the regex hit at face value, call it a hardcoded credential, and stamp the project NO-GO over a value that doesn't exist. Runworthy files a placeholder in a template as informational, where it can't touch the grade. In the graded report it never comes up at all.

Run it

pip install "runworthy[llm]"
runworthy doctor
runworthy scan <repo-url-or-path>

doctor lists the three open-source detectors and tells you which are missing. --no-llm runs free with no key and no model; the graded report runs on your own key — Anthropic direct, or OpenRouter with one extra env var (RUNWORTHY_MODEL_TRANSPORT=openai_compat). The graded scans above cost less than a dime each, and the model sees redacted finding summaries (file, line, what the detector saw), never your source or secret values. The two repos here are Node and Python; the detectors cover the usual ecosystems either way.

The rubric behind the grade, the Agent Flight Rules, is CC BY 4.0: ten non-negotiables and nineteen supporting controls, yours whether or not you ever run the scanner.

A Runworthy scan reports findings and gaps from static analysis and your answers; it is not a certification, and it can't see how an agent behaves at runtime. Verify anything marked "likely" or "couldn't determine" before you rely on it.
